# Vionix 规则引擎设计方案

自建轻量规则引擎，支持设备数据的实时条件检测、定时巡检、告警管理与动作分发。规则、告警和设备分组属于租户内业务数据，必须贯穿 `tenant_id`。

> 实现级详细设计见：[开发详细设计说明书](../docs/development/03-详细设计说明书.md) 和 [数据与存储设计说明书](../docs/development/04-数据与存储设计说明书.md)。若实现文档中的 DDL 与本文不一致，以本文的租户隔离、索引和条件组合约束为准，并在实现前同步修订。

## 1. 核心能力

| 能力 | 说明 |
|------|------|
| 触发条件 | 组内 AND、组间 OR、持续时间、变化率 |
| 执行动作 | 平台告警、MQTT 下发、HTTP 回调、IM 推送 |
| 规则范围 | 单设备、设备分组、租户全局 |
| 执行模式 | 实时 MQTT 消费、定时 InfluxDB 巡检 |
| 告警策略 | 频率抑制、恢复通知、告警升级 |
| 多租户 | 所有规则、分组、告警和动作日志按 `tenant_id` 隔离 |

## 2. 架构

```
MQTT 消息
  -> 解析 tenant_id + device_id + metric fields
  -> Spring Boot MQTT Consumer
  -> RuleEngine 按 tenant_id 和 scope 定位候选规则
  -> StateTracker 判断持续时间/变化率
  -> AlertManager 抑制、恢复、升级
  -> ActionDispatcher 执行动作
  -> WebSocket 按租户和订阅推送告警

定时巡检：
  @Scheduled
  -> 按租户查询启用规则
  -> 查询 InfluxDB 聚合数据
  -> RuleEngine 评估
  -> AlertManager + ActionDispatcher
```

## 3. 实时消息输入模型

规则引擎消费的实时消息必须已经解析出以下字段：

| 字段 | 来源 | 必选 | 说明 |
|------|------|------|------|
| `tenant_id` | MQTT topic、设备注册表或认证上下文 | 是 | 规则隔离边界 |
| `device_id` | MQTT topic 或 payload | 是 | DEVICE/GROUP 规则匹配主键 |
| `source` | payload 或输入通道 | 是 | `mqtt`、`modbus`、`http` |
| `timestamp` | payload 或接收时间 | 是 | 持续时间和变化率窗口使用 |
| `metrics` | payload 数值字段 | 是 | 如 `temperature`、`humidity` |

规则引擎不得依赖前端过滤来隔离租户或设备；候选规则定位必须在服务端完成。

## 4. 数据模型

规则引擎表存储在 MySQL。所有业务表都直接携带 `tenant_id`，便于 MyBatis 多租户拦截器统一追加过滤条件，也便于建立租户内唯一索引。

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| `device_group` | 设备分组 | `tenant_id`, `name`, `description`, `created_by`, `status` |
| `device_group_member` | 分组设备关联 | `tenant_id`, `group_id`, `device_id` |
| `rule` | 规则主表 | `tenant_id`, `name`, `scope`, `target_id`, `severity`, `suppress_minutes`, `schedule_cron`, `enabled`, `created_by` |
| `rule_condition` | 规则条件 | `tenant_id`, `rule_id`, `group_index`, `condition_type`, `metric`, `operator`, `threshold`, `duration_seconds`, `rate_window_seconds` |
| `rule_action` | 规则动作 | `tenant_id`, `rule_id`, `action_type`, `config`, `enabled` |
| `alert` | 告警记录 | `tenant_id`, `rule_id`, `device_id`, `severity`, `status`, `trigger_value`, `fired_at`, `resolved_at`, `escalated_at` |
| `alert_action_log` | 动作执行日志 | `tenant_id`, `alert_id`, `action_type`, `request`, `response`, `success`, `created_at` |

### 4.1 租户索引和唯一约束

| 表 | 约束/索引 |
|----|-----------|
| `device_group` | `UNIQUE KEY uk_tenant_group_name (tenant_id, name)` |
| `device_group_member` | `UNIQUE KEY uk_tenant_group_device (tenant_id, group_id, device_id)` |
| `rule` | `INDEX idx_tenant_scope_target (tenant_id, scope, target_id, enabled)` |
| `rule_condition` | `INDEX idx_rule_condition_rule (tenant_id, rule_id, group_index)` |
| `rule_action` | `INDEX idx_rule_action_rule (tenant_id, rule_id)` |
| `alert` | `INDEX idx_alert_tenant_status (tenant_id, status, fired_at)`, `INDEX idx_alert_rule_status (tenant_id, rule_id, status)` |
| `alert_action_log` | `INDEX idx_action_log_alert (tenant_id, alert_id, created_at)` |

即使实现阶段不启用数据库外键，也必须在服务层定义级联清理策略：删除规则时删除条件、动作，并将未恢复告警按业务规则关闭或保留审计；删除告警时保留动作日志还是级联删除需由产品审计要求决定，默认保留日志。

### 4.2 条件组合语义

当前版本只支持“组内 AND、组间 OR”：

```
(group_index = 0 的所有条件 AND)
OR
(group_index = 1 的所有条件 AND)
OR
...
```

示例：

```
(temperature > 80 持续 5 分钟) OR (temperature 1 分钟内上升 > 10)
```

实现约束：

- `rule_condition` 不使用 `logic_op` 表达组间关系。
- 前端编辑器、后端校验器、规则编译器和数据库模型必须共享同一语义。
- 如果后续需要表达嵌套条件，如 `(A AND B) OR (C AND (D OR E))`，应新增条件树 JSON 或表达式 AST 字段，不在 `group_index` 上继续叠加半表达逻辑。

## 5. 核心组件

| 组件 | 职责 |
|------|------|
| RuleEngine | 启动时按租户加载启用规则并编译为内存结构；按 `tenant_id + scope + target_id` 建立索引；规则变更时刷新对应租户缓存 |
| StateTracker | 按 `tenant_id + device_id + metric` 维护滑动窗口；支持持续时间判断和变化率计算 |
| AlertManager | 按 `tenant_id + rule_id + device_id` 维护告警状态；执行抑制、恢复和升级 |
| ActionDispatcher | 异步执行规则动作；动作配置中的 URL、Token 等敏感字段按安全方案加密存储或引用 Secret |
| RuleScheduler | 定时巡检启用的 `schedule_cron` 规则，查询 InfluxDB 聚合数据后交给 RuleEngine |

## 6. API 设计

所有 API 都从 JWT 中解析 `tenant_id`，普通租户用户不能通过请求参数切换租户。超级管理员跨租户操作需要显式 `X-Tenant-Id`，并记录审计日志。

### 6.1 规则管理

```
POST   /api/rules              创建规则（含 conditions + actions）
PUT    /api/rules/{id}          更新规则
DELETE /api/rules/{id}          删除规则
PUT    /api/rules/{id}/toggle   启用/禁用规则
GET    /api/rules               分页查询规则列表
GET    /api/rules/{id}          规则详情
```

租户边界：

- 创建规则时服务端写入当前 `tenant_id`。
- 查询列表自动追加 `tenant_id = currentTenant`。
- `scope=DEVICE` 时 `target_id=device_id` 必须属于当前租户。
- `scope=GROUP` 时 `target_id=group_id` 必须属于当前租户。
- 租户内规则名称唯一建议使用 `(tenant_id, name)`。

### 6.2 告警查询

```
GET /api/alerts            分页查询告警记录
GET /api/alerts/firing     当前正在触发的告警
GET /api/alerts/stats      告警统计（按级别/设备/时间段聚合）
```

查询参数：`severity`、`status`、`deviceId`、`startTime`、`endTime`、`pageNum`、`pageSize`。所有查询都必须附加当前租户过滤。

### 6.3 设备分组管理

```
POST   /api/device-groups               创建分组
PUT    /api/device-groups/{id}          更新分组
DELETE /api/device-groups/{id}          删除分组
GET    /api/device-groups               分组列表
POST   /api/device-groups/{id}/devices  绑定设备到分组
DELETE /api/device-groups/{id}/devices  从分组移除设备
```

绑定设备时必须校验 `device_id` 属于当前租户，避免跨租户把设备加入分组。

## 7. 告警实时推送

告警推送复用 WebSocket 通道，但服务端必须按订阅和权限过滤，不能广播全量告警后让前端自行过滤。

握手鉴权：

```
WebSocket /ws
Authorization: Bearer {accessToken}
```

使用 STOMP 时，也可以在 `CONNECT` frame 中携带 Authorization header。服务端解析 Token 后把 `tenant_id`、`user_id`、权限集合绑定到连接会话。

订阅路径：

```
/topic/tenant/{tenantId}/alerts
/topic/tenant/{tenantId}/device/{deviceId}/alerts
```

订阅校验：

- `{tenantId}` 必须等于当前会话租户，超级管理员除外。
- 订阅设备级告警时必须校验用户是否有该设备的数据权限。
- 前端 `$device` 过滤只用于展示层，不能作为安全边界。

告警消息：

```json
{
  "type": "ALERT_FIRING",
  "tenantId": 1,
  "data": {
    "alertId": 123,
    "ruleName": "高温告警",
    "severity": "CRITICAL",
    "deviceId": "sensor-001",
    "triggerValue": 92.5,
    "firedAt": "2026-05-26T10:01:00Z"
  }
}
```

`ALERT_RESOLVED` 和 `ALERT_ESCALATED` 使用同一 envelope，`type` 区分事件类型。
