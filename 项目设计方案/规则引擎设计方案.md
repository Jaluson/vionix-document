# Vionix 规则引擎设计方案

自建轻量规则引擎，支持设备数据的实时条件检测、定时巡检、告警管理与动作分发。

> 完整设计文档：[规则引擎详细设计](../docs/superpowers/specs/2026-05-26-rule-engine-design.md)

## 1. 核心能力

| 能力 | 说明 |
|------|------|
| 触发条件 | 复合条件（AND/OR 分组）+ 持续时间 + 变化率 |
| 执行动作 | 平台告警、MQTT 下发、HTTP 回调、IM 推送（可多选组合） |
| 规则范围 | 单设备 + 设备分组 + 全局 |
| 执行模式 | 实时（MQTT 消费）+ 定时巡检（InfluxDB 查询） |
| 告警策略 | 频率抑制 + 恢复通知 + 告警升级 |

## 2. 架构

```
MQTT 消息 ──→ Spring Boot MQTT Consumer
                    │
                    ▼
            ┌──────────────┐
            │ RuleEngine   │ ← 内存缓存编译后规则 + 匹配索引
            │ (核心引擎)    │
            └──────┬───────┘
                   │ 匹配成功
                   ▼
            ┌──────────────┐
            │ AlertManager │ ← 告警抑制/恢复/升级状态管理
            └──────┬───────┘
                   │
                   ▼
            ┌──────────────┐
            │ActionDispatcher│
            └──┬───┬───┬──┬─┘
               │   │   │  │
               ▼   ▼   ▼  ▼
            Alert  MQTT HTTP IM
            记录   下发 回调 推送
              │
              ▼
           WebSocket → 前端实时告警

定时巡检（@Scheduled）：
  InfluxDB 查询 → RuleEngine 评估 → AlertManager → ActionDispatcher
```

## 3. 数据模型

7 张表存储在 MySQL，DDL 详见 [规则引擎详细设计](../docs/superpowers/specs/2026-05-26-rule-engine-design.md) 第 6 节。

| 表名 | 说明 | 核心字段 |
|------|------|---------|
| `device_group` | 设备分组 | name, description |
| `device_group_member` | 分组设备关联 | group_id, device_id |
| `rule` | 规则主表 | name, scope(DEVICE/GROUP/GLOBAL), severity, suppress_minutes, schedule_cron |
| `rule_condition` | 规则条件 | condition_type(THRESHOLD/RATE), metric, operator, threshold, duration_seconds, rate_window_seconds |
| `rule_action` | 规则动作 | action_type(ALERT/MQTT/HTTP/IM), config(JSON) |
| `alert` | 告警记录 | rule_id, device_id, severity, status(FIRING/RESOLVED/ESCALATED), trigger_value |
| `alert_action_log` | 动作执行日志 | alert_id, action_type, request, response, success |

**条件组合逻辑**：同 `group_index` 内 AND，不同 `group_index` 间 OR。

示例：`(温度>80 持续5分钟) OR (温度1分钟内上升>10度)` → 两个条件分别在 group_index 0 和 1。

## 4. 核心组件

| 组件 | 职责 |
|------|------|
| **RuleEngine** | 启动时从 MySQL 加载规则编译为内存缓存；按 scope 建立匹配索引（GLOBAL/DEVICE/GROUP），MQTT 消息到达时快速定位候选规则评估；规则变更时刷新缓存 |
| **StateTracker** | 每设备每指标维护滑动窗口（环形缓冲区，600s）；支持持续时间判断（`duration_seconds > 0`）和变化率计算（`condition_type = RATE`） |
| **AlertManager** | 维护告警状态映射；频率抑制（suppress_minutes 窗口内不重复触发）；恢复检测（条件不满足时 RESOLVED + 恢复通知）；告警升级（FIRING 超 escalation_minutes 自动 ESCALATED） |
| **ActionDispatcher** | 异步并行执行规则关联的所有动作；每个动作类型有独立 Executor；失败重试 1 次，记录到 alert_action_log |

**双模式执行**：

- **实时模式**：MQTT Consumer 收到消息 → RuleEngine 即时评估 → AlertManager → ActionDispatcher
- **定时巡检**：`@Scheduled` 按 `schedule_cron` 查询 InfluxDB 做评估，适用于设备离线检测、聚合指标规则

## 5. API 设计

### 5.1 规则管理

```
POST   /api/rules              创建规则（含 conditions + actions）
PUT    /api/rules/{id}          更新规则
DELETE /api/rules/{id}          删除规则
PUT    /api/rules/{id}/toggle   启用/禁用规则
GET    /api/rules               分页查询规则列表
GET    /api/rules/{id}          规则详情
```

### 5.2 告警查询

```
GET /api/alerts            分页查询告警记录
GET /api/alerts/firing     当前正在触发的告警
GET /api/alerts/stats      告警统计（按级别/设备/时间段聚合）
```

查询参数：`severity`、`status`、`deviceId`、`startTime`、`endTime`、`pageNum`（默认 1）、`pageSize`

### 5.3 设备分组管理

```
POST   /api/device-groups              创建分组
PUT    /api/device-groups/{id}         更新分组
DELETE /api/device-groups/{id}         删除分组
GET    /api/device-groups              分组列表
POST   /api/device-groups/{id}/devices 绑定设备到分组
DELETE /api/device-groups/{id}/devices 从分组移除设备
```

## 6. 告警实时推送

复用现有 WebSocket 通道（`/ws`），新增告警消息类型：

```json
// 告警触发
{ "type": "ALERT_FIRING", "data": { "alertId": 123, "ruleName": "高温告警", "severity": "CRITICAL", "deviceId": "sensor-001", "triggerValue": 92.5 }}

// 告警恢复
{ "type": "ALERT_RESOLVED", "data": { "alertId": 123, "ruleName": "高温告警", "resolveTime": "..." }}

// 告警升级
{ "type": "ALERT_ESCALATED", "data": { "alertId": 123, "escalateTime": "..." }}
```
