# 项目设计方案审查报告

审查对象：`项目设计方案/` 下 8 份设计文档，并对照当前仓库中的 `docker-compose.yml`、`telegraf/telegraf.conf`、`influxdb/init.sh`、`mosquitto/config/mosquitto.conf`。

审查结论：文档覆盖了 IoT 监控平台的主要模块，但目前更像“目标架构草案”，还不能直接作为实施基线。主要问题集中在：设计与当前仓库不一致、时序降采样口径错误、多租户字段未贯穿业务模型、安全设计复杂但缺少部署依赖、实时数据模型没有定义清楚。

## 高优先级问题

### 1. 部署方案宣称完整编排，但当前仓库不可按文档一键启动完整系统

证据：

- `项目设计方案/基础设施与部署方案.md:10` 到 `:12` 将 MySQL、Backend、Frontend 列为完整服务组件。
- 同文档 `:25` 到 `:31` 要求存在 `mysql/init.sql`、`backend/`、`frontend/`。
- 同文档 `:180` 到 `:237` 给出了 MySQL、Backend、Frontend 的 Compose 配置。
- 当前 `docker-compose.yml:3` 到 `:40` 只有 Mosquitto、InfluxDB、Telegraf，缺少 MySQL、Backend、Frontend。

影响：按文档执行 `docker-compose up -d --build` 不能得到文档承诺的 API、前端、规则引擎和 RBAC 能力。设计文档和交付物状态不一致，会误导开发排期和验收。

建议：

- 将部署方案拆成“当前可运行最小环境”和“目标完整环境”两节。
- 补齐或移除 `mysql/`、`backend/`、`frontend/` 的交付承诺。
- 如果当前阶段只交付基础设施，文档中的启动命令和访问地址应明确标注哪些不可用。

### 2. InfluxDB 降采样链路会生成错误字段，并导致小时/天级统计口径失真

证据：

- `项目设计方案/InfluxDB存储设计方案.md:14` 到 `:27` 规定每级存 `temperature_mean/sum/max/min/count`。
- `influxdb/init.sh:47` 到 `:51` 从 `device_raw` 生成 `*_mean/*_sum/*_max/*_min/*_count`，这一级基本符合预期。
- `influxdb/init.sh:64` 到 `:75` 从 `device_min` 再对所有字段聚合并再次追加后缀，会生成 `temperature_mean_mean`、`temperature_sum_sum` 等二次后缀。
- `influxdb/init.sh:85` 到 `:96` 天级同样会继续叠加后缀。

影响：

- 后端按 `fields + agg` 拼接 `temperature_mean` 查询小时级时，实际数据可能是 `temperature_mean_mean`。
- 从分钟均值再求小时均值没有按 `count` 加权，数据点缺失时会产生错误平均值。
- `sum/count/max/min` 也不能对所有上一级字段一概重复聚合。

建议：

- 明确每级 Task 的输入字段和输出字段映射。
- 小时/天级均值应使用 `sum(value_sum) / sum(value_count)`，最大值取 `max(value_max)`，最小值取 `min(value_min)`，总和取 `sum(value_sum)`，计数取 `sum(value_count)`。
- Task 中应过滤字段后分别计算，避免后缀连乘。

### 3. 多租户设计要求所有业务表有 `tenant_id`，但规则引擎模型没有租户字段

证据：

- `项目设计方案/RBAC设计方案.md:38` 到 `:44` 规定共享数据库 + `tenant_id` 行级隔离，且所有业务表携带 `tenant_id`。
- `项目设计方案/规则引擎设计方案.md:55` 到 `:61` 的规则、告警、设备分组表摘要没有 `tenant_id`。
- 详细 DDL 在 `docs/superpowers/specs/2026-05-26-rule-engine-design.md:328` 到 `:399` 创建 `device_group`、`rule`、`alert` 等表，同样没有 `tenant_id`。

影响：RBAC 上线后，规则、告警、设备分组无法做租户隔离。MyBatis 多租户拦截器也无法对没有 `tenant_id` 的表追加过滤条件。

建议：

- 给 `device_group`、`device_group_member`、`rule`、`rule_condition`、`rule_action`、`alert`、`alert_action_log` 增加租户隔离策略。
- 子表可直接冗余 `tenant_id`，便于统一拦截和索引；或明确哪些表通过父表 Join 隔离，并说明拦截器如何处理。
- 所有规则/告警 API 必须在请求、查询条件和唯一索引中体现租户边界。

### 4. 设备标识和 InfluxDB tag 模型没有定义，前端过滤和规则匹配缺少基础

证据：

- `项目设计方案/Web监控前端方案.md:24` 到 `:29` 设计 `/api/metrics` 支持 `device` 作为 tag 过滤。
- `telegraf/telegraf.conf:11` 到 `:18` 只配置了 MQTT JSON 写入和 `name_override`，没有定义从 topic 或 payload 提取 `device_id` tag。
- `项目设计方案/规则引擎设计方案.md:71` 依赖 `DEVICE/GROUP/GLOBAL` 规则索引，实时消息必须能稳定解析 `device_id`。

影响：同一个 measurement 下如果没有稳定 tag，后端无法按设备查询历史数据，规则引擎也无法可靠匹配单设备和设备组规则。

建议：

- 定义统一上报模型：topic 规范、payload 字段、必选 tag、字段类型。
- Telegraf 应明确 `topic_parsing` 或 payload tag 提取规则，保证 `device_id`、`tenant_id`、`source` 等维度进入 InfluxDB tag。
- 后端 WebSocket 消息结构也应和 InfluxDB 写入模型保持一致。

## 中优先级问题

### 5. 规则条件组合模型表达能力和字段语义不一致

证据：

- `项目设计方案/规则引擎设计方案.md:63` 规定同 `group_index` 内 AND、不同组之间 OR。
- 详细设计 `docs/superpowers/specs/2026-05-26-rule-engine-design.md:103` 到 `:104` 又定义 `logic_op` 为“与上一组的连接运算符”。

影响：摘要设计表示组间固定 OR，但数据模型保留 `logic_op`，实现时容易出现前后端、后端编译器和数据库语义不一致。该模型也无法表达 `(A AND B) OR (C AND (D OR E))` 这类嵌套条件。

建议：

- 如果只支持“组内 AND、组间 OR”，删除或忽略 `logic_op`，降低复杂度。
- 如果要支持 AND/OR 组合，应改为条件树 JSON 或表达式 AST，而不是靠 `group_index + logic_op` 半表达。

### 6. 安全设计存在过度加密和密钥管理风险

证据：

- `项目设计方案/RBAC设计方案.md:17` 已要求 HTTPS。
- 同文档 `:219` 到 `:287` 又要求 RSA 加密密码、手机号、邮箱、Webhook URL 等敏感传输字段。
- 同文档 `:223` 到 `:230` 要求 RSA 私钥固定写入 `application.yml`。

影响：

- HTTPS 下再做前端 RSA 加密对登录安全提升有限，却显著增加密钥轮换、前端兼容和调试复杂度。
- 私钥写入应用配置容易进入镜像、日志、配置仓库或运维备份，风险较高。
- 文档提到 AES 会话密钥，但后续流程没有完整设计，容易形成未实现的安全承诺。

建议：

- 优先采用 HTTPS + BCrypt/Argon2 + HttpOnly refresh cookie + CSRF/CORS 策略。
- 如果保留 RSA，只用于特定高敏字段，并明确密钥来源、轮换、吊销和泄露处理。
- 将密钥放入环境变量、Secret 管理或 KMS，不建议写入普通 `application.yml`。

### 7. Redis 被用于关键安全能力，但部署方案没有编排 Redis

证据：

- `项目设计方案/RBAC设计方案.md:213` 要求登出时将 accessToken 加入 Redis 黑名单。
- 同文档 `:712` 到 `:720` 列出 Token 黑名单、登录失败计数、权限缓存，并说 Redis 可选。
- `项目设计方案/基础设施与部署方案.md:5` 到 `:12` 的完整服务组件没有 Redis，Compose 示例也没有 Redis。

影响：如果 Redis 是可选的，登出立即失效、登录失败锁定、权限缓存这些能力需要明确本地缓存替代方案和多实例限制；否则多副本部署会出现状态不一致。

建议：

- 明确单实例开发环境可用 Caffeine，多实例/生产环境必须 Redis。
- 部署文档补充 Redis 服务和环境变量，或把依赖 Redis 的安全能力降级描述清楚。

### 8. WebSocket 通道缺少订阅、鉴权和租户隔离协议

证据：

- `项目设计方案/Web监控前端方案.md:81` 到 `:89` 只描述建立 WebSocket 后接收实时数据和告警。
- `项目设计方案/RBAC设计方案.md:17` 提到 WS Token 鉴权，但没有握手协议和订阅模型。
- `项目设计方案/低代码仪表盘设计方案.md:345` 要求前端按 `$device + metric` 过滤消息。

影响：如果后端广播全量消息，前端过滤会泄露其他设备或租户的数据，也会带来性能问题。告警和实时指标共用 `/ws` 时，需要 topic/type/订阅协议来避免消息混杂。

建议：

- 定义 WS 握手鉴权方式：Header、query token 或 STOMP CONNECT header。
- 定义订阅路径，如 `/topic/tenant/{tenantId}/device/{deviceId}/metrics`、`/topic/alerts`。
- 服务端必须按租户和权限过滤，前端过滤只能作为展示层逻辑。

## 低优先级问题

### 9. 项目名称拼写不一致

证据：

- 大多数文档使用 `Vionix`。
- `项目设计方案/RBAC设计方案.md:1`、`:5` 使用 `Voinix`。
- `项目设计方案/Web监控前端方案.md:50` 使用 `Voinix`。
- `docs/superpowers/specs/2026-05-26-rule-engine-design.md:1`、`:5` 使用 `Voinix`。

建议：统一为一个项目名，避免后续包名、页面标题、文档标题和品牌资产混乱。

### 10. 规则引擎 DDL 缺少外键、必要索引和生命周期字段

证据：

- 详细 DDL `docs/superpowers/specs/2026-05-26-rule-engine-design.md:328` 到 `:399` 基本只定义主键、少量唯一约束和告警索引。
- `rule_condition.rule_id`、`rule_action.rule_id`、`alert.rule_id`、`alert_action_log.alert_id` 没有外键或至少索引。

影响：规则详情查询、规则删除清理、告警动作日志查询都会受影响。缺少约束也容易留下孤儿数据。

建议：

- 即使不使用数据库外键，也应补充索引和级联删除策略。
- 给 `rule_condition(rule_id)`、`rule_action(rule_id)`、`alert(rule_id, status)`、`alert_action_log(alert_id)` 建索引。

### 11. 低代码仪表盘说“不新增数据接口”，但实际新增了变量接口

证据：

- `项目设计方案/低代码仪表盘设计方案.md:14` 写“复用现有 WS + InfluxDB API，不新增数据接口”。
- 同文档 `:258` 到 `:259` 新增了 `GET /api/dashboard-vars/devices` 和 `GET /api/dashboard-vars/metrics`。

建议：改成“不新增监控数据查询接口，新增仪表盘元数据和变量选项接口”。

## 建议的整改顺序

1. 先冻结当前阶段边界：当前仓库到底交付基础设施、后端骨架，还是完整可运行平台。
2. 修正部署文档和 `docker-compose.yml` 的一致性。
3. 定义设备数据标准模型，包括 MQTT topic、payload、InfluxDB tag/field、WebSocket 消息结构。
4. 修正 InfluxDB 降采样 Task，避免二次后缀和错误聚合。
5. 将 RBAC 的 `tenant_id` 贯穿规则、告警、仪表盘、设备分组等业务表。
6. 收敛安全方案，明确 Redis、HTTPS、Token、RSA 的真实生产依赖。
7. 统一项目名称和 API/WS 协议细节。
