# Vionix 系统架构设计方案

## 1. 项目概述

Vionix 是 IoT 设备监控平台，支持多数据源采集、多粒度时序存储、实时监控、历史查询、规则告警、RBAC 多租户隔离和低代码仪表盘。

当前仓库阶段：已具备 Mosquitto、InfluxDB、Telegraf 的最小基础设施。

目标完整阶段：补齐 MySQL、Redis、Spring Boot 4 Backend、Vue Frontend 后，形成完整可运行平台。

## 2. 当前最小架构

```
设备(MQTT JSON)
  -> Mosquitto
  -> Telegraf
  -> InfluxDB device_raw
  -> InfluxDB Task 降采样到 device_min/device_hour/device_day
```

当前能力：

- MQTT Broker：`localhost:1883`
- InfluxDB 管理界面：`http://localhost:8086`
- Telegraf MQTT JSON 写入
- Bucket 和 Task 初始化

当前未交付：

- 后端 API
- WebSocket 实时推送
- RBAC 和规则引擎
- 前端页面和低代码仪表盘

## 4. 核心数据流

### 4.1 设备数据写入

```
MQTT topic: vionix/{tenant_id}/{device_id}/metrics
  -> Mosquitto
  -> Telegraf 或 Backend 写入网关
  -> 补齐 tenant_id、device_id、source tag
  -> InfluxDB device_raw
  -> Task 按固定字段名降采样
```

必选 tag：

| Tag | 说明 |
|-----|------|
| `tenant_id` | 租户隔离边界 |
| `device_id` | 设备过滤和规则匹配主键 |
| `source` | 数据来源 |

### 4.2 实时推送

```
MQTT Consumer
  -> 解析 tenant_id + device_id
  -> WebSocket/STOMP 服务端权限校验
  -> 推送到 /topic/tenant/{tenantId}/device/{deviceId}/metrics
```

服务端必须按租户和数据权限过滤，前端过滤只用于展示。

### 4.3 历史查询

```
GET /api/metrics
  -> JWT 解析 tenant_id
  -> 校验 deviceId 数据权限
  -> 根据时间范围选择 bucket
  -> 查询 InfluxDB 聚合字段
```

### 4.4 规则告警

```
实时规则：MQTT Consumer -> RuleEngine -> AlertManager -> ActionDispatcher
巡检规则：Scheduler -> InfluxDB 查询 -> RuleEngine -> AlertManager
告警推送：AlertManager -> WebSocket 订阅 topic
```

规则、分组、告警、动作日志均携带 `tenant_id`。

## 5. 技术选型

| 层 | 技术 | 说明 |
|----|------|------|
| 实时推送 | WebSocket/STOMP | 支持订阅路径、鉴权和服务端过滤 |
| MQTT 客户端 | Eclipse Paho / Spring Integration MQTT | 订阅 Mosquitto，同时供规则引擎实时评估 |
| 时序数据库 | InfluxDB 2.x | 原始数据和分钟/小时/天聚合 bucket |
| 时序采集 | Telegraf | MQTT/Modbus/HTTP 多源写入 |
| 关系数据库 | MySQL + MyBatis-Plus | RBAC、规则、告警、仪表盘元数据 |
| 缓存/安全状态 | Redis | Token 黑名单、登录失败计数、权限缓存；生产必需 |
| 规则引擎 | 自建内存引擎 | 编译缓存、匹配索引、滑动窗口状态跟踪 |
| 前端图表 | Vue + ECharts | 实时曲线、历史趋势、低代码仪表盘 |
| 容器化 | Docker Compose | 当前最小基础设施可运行，完整环境需补齐应用目录 |

## 6. 架构约束

- 当前仓库只承诺基础设施可启动；完整平台能力以补齐 Backend、Frontend、MySQL、Redis 后为准。
- 所有业务表必须具备租户隔离策略，规则和告警相关表直接携带 `tenant_id`。
- InfluxDB 所有历史查询必须基于 `tenant_id`、`device_id` 等 tag 过滤。
- WebSocket 订阅必须在服务端完成鉴权和数据权限校验。
- Redis 在生产和多实例环境中为必需组件；没有 Redis 时不能承诺 accessToken 登出立即全局失效。
