# Vionix 需求规格说明书

## 1. 项目目标

Vionix 是面向多租户 IoT 场景的设备监控平台，目标是提供设备数据接入、实时监控、历史趋势、规则告警、权限隔离和低代码仪表盘能力。

当前阶段目标是完成从最小基础设施到完整可用平台的研发交付：

```
Mosquitto + Telegraf + InfluxDB
  -> Spring Boot Backend
  -> Vue Frontend
  -> MySQL + Redis
  -> RBAC + 规则引擎 + WebSocket + 低代码仪表盘
```

## 2. 用户角色

| 角色 | 说明 | 核心诉求 |
|------|------|----------|
| 超级管理员 | 平台级管理者，管理租户和全局配置 | 创建租户、跨租户运维、系统配置 |
| 租户管理员 | 单租户内管理员 | 管理用户、角色、设备分组、规则和仪表盘 |
| 运维人员 | 设备监控和故障处理人员 | 查看实时数据、告警、历史趋势和设备状态 |
| 普通查看用户 | 只读用户 | 查看被授权设备和公开仪表盘 |
| 外部设备 | MQTT/Modbus/HTTP 数据生产端 | 按统一协议上报指标数据 |

## 3. 范围定义

### 3.1 本期范围

| 范围 | 内容 |
|------|------|
| 设备数据接入 | MQTT JSON 上报，统一解析 `tenant_id`、`device_id`、`source` 和指标字段 |
| 设备基础目录 | 维护租户内设备标识、名称、状态、来源和权限边界，支撑监控、规则、仪表盘变量和 WebSocket 订阅校验 |
| 时序存储 | InfluxDB 原始数据、分钟、小时、天级聚合存储 |
| 实时监控 | WebSocket/STOMP 推送秒级指标和告警事件 |
| 历史查询 | 按时间范围、设备、指标和聚合方式查询历史数据 |
| RBAC | 多租户、用户、角色、权限、数据范围、双 token 认证 |
| 规则引擎 | 规则条件、设备分组、告警状态、动作分发 |
| 低代码仪表盘 | 仪表盘 CRUD、布局 JSON、变量绑定、组件渲染 |
| 部署 | 本地开发、测试和生产部署基线 |

### 3.2 暂不纳入范围

| 内容 | 说明 |
|------|------|
| 大规模设备注册中心 | 本期只实现最小设备目录和权限边界；批量注册、固件、影子、物模型等完整设备生命周期后续拆分 |
| 多区域高可用集群 | 本期部署文档覆盖单区域生产基线，不覆盖跨地域容灾 |
| 复杂规则表达式 AST | 当前规则条件只支持“组内 AND、组间 OR” |
| 报表系统 | 可基于历史查询和仪表盘扩展，不作为本期核心交付 |

## 4. 功能需求

| 编号 | 需求 | 优先级 | 验收条件 |
|------|------|--------|----------|
| FR-01 | 设备通过 MQTT 上报 JSON 指标 | P0 | 平台能接收 `vionix/{tenant_id}/{device_id}/metrics` topic，解析 tag 和数值 field |
| FR-02 | 原始指标写入 InfluxDB | P0 | `device_raw` 中存在 `device_metrics`，包含 `tenant_id`、`device_id`、`source` tag |
| FR-03 | 指标自动降采样 | P0 | 数据按分钟、小时、天写入对应 bucket，字段不出现二次聚合后缀 |
| FR-04 | 查询历史指标 | P0 | `GET /api/metrics` 能按租户、设备、时间、字段、聚合函数返回数据 |
| FR-05 | 秒级实时监控 | P0 | 前端订阅设备 topic 后能收到当前租户内授权设备的实时点 |
| FR-06 | 用户登录和 token 刷新 | P0 | 支持 accessToken、HttpOnly refreshToken、登出和刷新 |
| FR-07 | 多租户数据隔离 | P0 | 普通租户用户无法访问其他租户用户、规则、告警、仪表盘和指标 |
| FR-08 | 权限控制 | P0 | API、菜单和数据范围均按角色权限生效 |
| FR-09 | 设备基础目录 | P0 | 租户内设备可创建、查询、启停，并以 `(tenant_id, device_id)` 唯一 |
| FR-10 | 设备分组管理 | P1 | 租户内可创建分组并绑定本租户设备 |
| FR-11 | 规则管理 | P1 | 可创建、更新、启停、删除规则，规则名称租户内唯一 |
| FR-12 | 规则实时评估 | P1 | MQTT 消息能触发规则计算并生成告警 |
| FR-13 | 告警管理 | P1 | 可查询触发中、历史告警和统计数据 |
| FR-14 | 告警实时推送 | P1 | 告警事件通过 WebSocket 按租户和设备权限推送 |
| FR-15 | 规则动作分发 | P2 | 支持 MQTT、HTTP 或即时消息动作，动作日志可查询 |
| FR-16 | 低代码仪表盘 CRUD | P1 | 用户可创建、编辑、删除、查看仪表盘 |
| FR-17 | 仪表盘变量绑定 | P1 | `$device`、`$timeRange` 等变量可驱动组件数据源 |
| FR-18 | 仪表盘实时组件 | P1 | 组件可复用 `/api/metrics` 和 WebSocket，不新增时序数据查询协议 |
| FR-19 | 审计和安全状态 | P1 | 登录失败计数、token 黑名单、权限缓存使用 Redis 支撑 |

## 5. 非功能需求

| 编号 | 类型 | 要求 |
|------|------|------|
| NFR-01 | 安全 | 所有受保护 API 必须校验 JWT，普通用户不能通过参数覆盖租户 |
| NFR-02 | 安全 | refreshToken 必须通过 HttpOnly、Secure、SameSite Cookie 下发 |
| NFR-03 | 安全 | 密钥、盐值、数据库密码不得进入普通配置文件和镜像 |
| NFR-04 | 性能 | 实时指标推送链路在正常负载下端到端延迟应小于 2 秒 |
| NFR-05 | 性能 | 常规历史查询应在合理索引和 bucket 路由下 3 秒内返回 |
| NFR-06 | 可维护性 | Backend、Frontend、数据库迁移和部署脚本必须独立可测试 |
| NFR-07 | 可观测性 | 关键服务需输出结构化日志、健康检查和基础指标 |
| NFR-08 | 可靠性 | 生产和多实例环境必须启用 Redis，保证安全状态共享 |
| NFR-09 | 数据质量 | 降采样均值必须按 count 加权，不能简单平均上一层均值 |
| NFR-10 | 兼容性 | `sensors/{device_id}` topic 仅允许单租户开发兼容，不作为生产协议 |
| NFR-11 | 可观测性 | API、MQTT 消费、规则评估、WS 推送、任务执行必须具备日志、指标和健康状态 |
| NFR-12 | 交付质量 | 主干合并和发布必须经过格式检查、测试、密钥扫描和文档同步检查 |

## 6. 业务规则

| 编号 | 规则 |
|------|------|
| BR-01 | 租户管理员只能管理本租户用户、角色、规则、告警、仪表盘和设备分组。 |
| BR-02 | 超级管理员跨租户操作必须显式指定目标租户并记录审计日志。 |
| BR-03 | 规则作用域为 `DEVICE` 时，`target_id` 表示 `device_id`；作用域为 `GROUP` 时，`target_id` 表示 `group_id`。 |
| BR-04 | 规则条件当前只支持组内 AND、组间 OR。 |
| BR-05 | 仪表盘 `layout_config` 中不得存储 `tenant_id`，租户由服务端从 JWT 注入。 |
| BR-06 | WebSocket 租户级订阅只能推送当前租户数据；设备级订阅必须校验设备数据权限。 |
| BR-07 | 生产环境中，指标写入、规则目标、设备分组绑定和仪表盘 `$device` 变量都必须引用当前租户内已登记且启用的设备。 |

## 7. 验收基线

| 验收项 | 通过标准 |
|--------|----------|
| 基础设施 | 当前 Compose 可启动 Mosquitto、Telegraf、InfluxDB，并能验证 MQTT 写入 |
| 数据链路 | 从 MQTT 到 InfluxDB 原始 bucket 和聚合 bucket 的数据链路可验证 |
| API | OpenAPI 或接口文档与实现一致，关键 API 有集成测试 |
| 权限 | 多租户和 RBAC 有跨租户越权测试 |
| 设备目录 | 设备归属、启停、分组绑定和跨租户访问校验通过 |
| 实时能力 | WebSocket 鉴权、订阅、取消订阅、重连和服务端过滤可验证 |
| 前端 | 核心页面具备加载、异常、空数据和权限隐藏状态 |
| 部署 | 测试环境可通过目标 Compose 或等价环境启动完整平台 |
| 质量 | 单元、集成、端到端、安全关键用例、密钥扫描和文档同步检查通过后才能发布 |
