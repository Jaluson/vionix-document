# Vionix 开发文档总览

本文档集基于 `docs/design/` 下的设计方案和 `docs/review/项目设计方案审查报告.md` 整理，用作 Vionix 后续研发、测试、部署和验收的统一工程基线。

## 适用范围

当前仓库已具备 Mosquitto、Telegraf、InfluxDB 的最小基础设施；Backend、Frontend、MySQL、Redis、RBAC、规则引擎、WebSocket 和低代码仪表盘仍属于目标完整平台的待开发内容。

本文档集描述目标完整平台的开发要求，同时明确当前阶段可运行边界，避免把未交付应用能力误认为当前仓库已经具备。

## 统一工程约束

| 编号 | 约束 |
|------|------|
| C-01 | 所有租户业务数据必须以 `tenant_id` 作为服务端隔离边界，普通用户不能通过请求参数切换租户。 |
| C-02 | 设备标识在 MQTT、InfluxDB、WebSocket、规则引擎中统一使用 `tenant_id + device_id`。 |
| C-03 | InfluxDB 聚合层字段统一为 `*_mean/*_sum/*_max/*_min/*_count`，不得出现二次后缀。 |
| C-04 | WebSocket 必须在服务端完成 token、租户、设备和数据权限校验。 |
| C-05 | Redis 是生产和多实例环境必需组件，用于 token 黑名单、登录失败计数和权限缓存。 |
| C-06 | 数据库密码、JWT 密钥、token hash 盐和私钥不得写入普通配置文件或 Compose 明文。 |
| C-07 | 当前仓库 Compose 只承诺基础设施可运行，完整平台需补齐 Backend、Frontend、MySQL、Redis 后再纳入一键编排。 |
| C-08 | 生产环境中的规则目标、WS 订阅、仪表盘变量和历史查询必须引用当前租户设备目录中的启用设备。 |
| C-09 | 主干合并和发布必须通过测试、文档同步检查和密钥扫描。 |

## 维护规则

1. 设计文档变更后，应同步检查本目录中需求、接口、数据和测试文档是否需要更新。
2. 接口新增或变更必须先更新 [05-接口协议说明书](05-接口协议说明书.md)，再进入开发。
3. 数据表或 InfluxDB 字段变更必须同步更新 [04-数据与存储设计说明书](04-数据与存储设计说明书.md) 和迁移脚本。
4. 每个功能开发任务必须能追溯到需求编号和验收用例。
5. 任何绕过 `tenant_id`、权限校验、服务端 WebSocket 过滤或密钥管理要求的实现，都不能进入主干。
6. 任何改变设备目录、设备状态或数据权限语义的实现，都必须同步更新接口、前端和测试文档。
