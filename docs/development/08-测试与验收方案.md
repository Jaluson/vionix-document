# Vionix 测试与验收方案

## 1. 测试目标

测试需要证明平台在功能、租户隔离、时序数据正确性、实时推送、安全状态和部署可用性上满足开发基线。重点风险是跨租户越权、降采样错误、WebSocket 过度广播、token 状态不一致和配置密钥泄露。

## 2. 测试分层

| 层级 | 范围 | 工具建议 |
|------|------|----------|
| 单元测试 | Service、规则计算、字段映射、权限判断 | JUnit、Mockito |
| 集成测试 | MySQL、Redis、InfluxDB、MQTT、WebSocket | Testcontainers 或测试 Compose |
| API 测试 | HTTP 接口、认证、分页、错误码 | REST Assured/Postman/Newman |
| 前端测试 | 组件、路由守卫、store、图表状态 | Vitest、Vue Test Utils |
| E2E 测试 | 登录、监控、规则、告警、仪表盘 | Playwright |
| 安全测试 | 越权、token、XSS、密钥 | 自动化用例 + 手工检查 |
| 性能测试 | 指标写入、历史查询、WS 推送 | k6/JMeter |

## 3. 关键测试用例

### 3.1 设备目录

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-DEVICE-001 | 租户 A 创建设备 `sensor-001` | 创建成功，`tenant_id + device_id` 唯一 |
| TC-DEVICE-002 | 租户 A 重复创建设备 `sensor-001` | 返回 409 |
| TC-DEVICE-003 | 租户 B 查询租户 A 设备详情 | 403 或 404，不返回设备数据 |
| TC-DEVICE-004 | 禁用设备后订阅 WS 或创建规则目标 | 操作被拒绝 |
| TC-DEVICE-005 | 仪表盘变量查询设备选项 | 只返回当前用户可见且启用设备 |

### 3.2 设备数据链路

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-METRIC-001 | 发布 `vionix/1/sensor-001/metrics` JSON | InfluxDB raw 写入 `tenant_id=1`、`device_id=sensor-001` |
| TC-METRIC-002 | 写入 temperature/humidity 数值 | 聚合 bucket 生成对应 `*_mean/*_sum/*_max/*_min/*_count` |
| TC-METRIC-003 | 查询 `level=min&fields=temperature&agg=mean` | 返回 `temperature_mean` 对应数据 |
| TC-METRIC-004 | 构造不均匀采样数据 | hour/day 均值按 count 加权 |
| TC-METRIC-005 | 查询其他租户 deviceId | 返回 403 或空结果，不返回数据 |

### 3.3 RBAC 和租户隔离

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-AUTH-001 | 正确用户名密码登录 | 返回 accessToken，refreshToken 写入 HttpOnly Cookie |
| TC-AUTH-002 | 登出后调用受保护 API | accessToken 被 Redis 黑名单拒绝 |
| TC-AUTH-003 | 租户 A 用户请求租户 B 用户详情 | 403 或 404，不泄露 B 数据 |
| TC-AUTH-004 | 无权限用户创建规则 | 403 |
| TC-AUTH-005 | 多实例下登出 | 所有实例均拒绝旧 token |
| TC-AUTH-006 | 超级管理员跨租户操作或修改权限 | 写入 `audit_log`，包含 tenant、user、action、resource、traceId |

### 3.4 WebSocket

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-WS-001 | 无 token 连接 `/ws` | 连接拒绝 |
| TC-WS-002 | 租户 A 订阅租户 B topic | 订阅拒绝 |
| TC-WS-003 | 订阅未授权设备 topic | 订阅拒绝 |
| TC-WS-004 | 设备产生指标 | 只有授权订阅者收到 `METRIC_POINT` |
| TC-WS-005 | token 过期后重连 | 前端 refresh 成功后恢复订阅 |

### 3.5 规则和告警

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-RULE-001 | 创建 DEVICE 规则，targetId 属于当前租户 | 创建成功 |
| TC-RULE-002 | 绑定其他租户 deviceId | 创建失败 |
| TC-RULE-003 | 组内多条件、组间多组 | 按 AND/OR 语义触发 |
| TC-RULE-004 | 抑制窗口内重复触发 | 不重复生成等价告警 |
| TC-RULE-005 | 告警触发 | MySQL 写告警，WebSocket 推送告警事件 |
| TC-RULE-006 | 配置 HTTP/MQTT 动作并触发规则 | 动作异步执行，成功或失败均记录 `alert_action_log` |

### 3.6 仪表盘

| 用例 | 步骤 | 期望 |
|------|------|------|
| TC-DASH-001 | 创建包含 `$device` 变量的仪表盘 | `layout_config` 保存成功，不包含 `tenant_id` |
| TC-DASH-002 | 非创建者编辑私有仪表盘 | 403，租户管理员除外 |
| TC-DASH-003 | 公开仪表盘查看 | 同租户授权用户可查看 |
| TC-DASH-004 | 组件请求数据 | 复用 `/api/metrics` 或 WebSocket |
| TC-DASH-005 | 非法组件 type | 后端或前端 schema 校验拒绝 |

## 5. 质量门禁

| 阶段 | 门禁 |
|------|------|
| 提交前 | 单元测试通过，格式检查通过，无密钥提交 |
| 合并前 | 相关集成测试通过，接口文档同步更新 |
| 测试环境发布 | 数据库迁移成功，冒烟测试通过 |
| 生产发布 | 安全关键用例通过，密钥扫描通过，备份完成，可回滚 |

## 6. 冒烟测试清单

1. 服务健康检查全部通过。
2. 管理员登录成功。
3. 创建租户、用户、角色成功。
4. 创建设备并在监控页设备列表可见。
5. 发布 MQTT 测试消息后能查询到时序数据。
6. 实时监控页面能收到 WebSocket 点位。
7. 创建规则后能触发告警。
8. 告警中心能看到触发记录。
9. 创建仪表盘并添加实时组件成功。
10. 登出后旧 token 失效。
11. 跨租户访问被拒绝。
